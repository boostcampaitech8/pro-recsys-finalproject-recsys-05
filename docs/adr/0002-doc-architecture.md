# ADR-0002 · 유지보수 문서 아키텍처

- **상태**: Accepted (2026-07-07)
- **맥락**: "지금 무슨 결정을 내리고 무슨 일을 하는지"를 `CLAUDE.md`에 계속 누적하면 비대해지고, 여러 브랜치에서 편집 시 충돌한다. 특히 진행상태(step·티켓 status)는 변동성이 높아 `feature→dev→main` 병합에서 충돌을 일으킨다. 문서를 **변동성·대상별로 분리**한다.

## 결정
문서 책임 분리:

| 문서 | 역할 | 변동성 | 대상 |
|---|---|---|---|
| `CLAUDE.md` | AI 세션 헌법·라우터(불변식·진입점) | 거의 고정 | AI |
| `docs/PRD.md` | 제품 방향성·요구사항(티켓 정합성 기준) | 낮음 | 팀 |
| `docs/adr/*.md` | 설계 결정 기록(MADR 경량) | 결정마다 추가 | 팀 |
| `docs/MAINTENANCE.md` | 하네스 구조 + 티켓 **정의**(durable) | 중 | 팀+AI |
| **GitHub Issues/Projects** | **라이브 진행상태**(status·현재 step) | 높음 | 팀 |

- 라이브 상태를 **버전관리 밖(Issues)**으로 빼서 브랜치 충돌을 원천 제거하고 팀 공유를 자연스럽게 한다.
- `docs/MAINTENANCE.md`의 step 보드/`[status]`는 **예시·초기값**이며, 권위 있는 진행상태는 Issues.

## 결과
- 새 결정 → 새 ADR(재작성 금지, `Superseded`로 대체).
- `CLAUDE.md`는 이 분리를 진입점으로만 안내한다.
- **트레이드오프**: 문서 수가 늘지만 각 문서가 단일 책임을 갖는다.
